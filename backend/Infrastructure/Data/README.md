# Infrastructure/Data

Database access layer. Contains the EF Core context that defines the schema and
maps the domain entities to PostgreSQL.

## Contents

**`AppDbContext`** — the single source of truth for the relational schema. Its
`OnModelCreating` configures every column constraint (lengths, required/optional),
the unique indexes (`User.Email`, `User.Handle`), the composite index backing the
public-by-handle query (`Project.OwnerId, IsPublished`), and the cascade-delete
relationship from user to projects. Migrations are scaffolded _from_ this
configuration — change the schema here, then generate a migration, never the
reverse.

## The Migrations folder (sibling)

`Migrations/` is a generated, append-only ledger of schema changes. **Do not
hand-edit it.** The `*.Designer.cs` files and `AppDbContextModelSnapshot.cs` are
marked `// <auto-generated />` and are rewritten by EF on every scaffold; the
plain migration files are regenerable too. The _narrative_ of how the schema
evolved (why ownership arrived when it did, why an index was later replaced by a
composite) lives in the top-level `ARCHITECTURE.md`, not in per-file comments.

## Provider notes

- **PostgreSQL via Npgsql.** Identity columns use Postgres `IdentityByDefault`.
- The connection string (`DefaultConnection`) is supplied by configuration and is
  required at startup — its absence is a fail-fast error, not a fallback. In
  development it's resolved from user-secrets; see the backend README for the
  per-environment setup.
